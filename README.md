# Discovering Visual Attributes in T2I Reward Models

This repository implements an iterative evolutionary search algorithm that discovers
visual attributes amplified during Best-of-N (BoN) sampling under a given
text-to-image (T2I) reward model — e.g., **ImageReward**, **PickScore**, **HPSv3**.
The method is grounded in the BoN amplification identity
(https://arxiv.org/html/2602.01002v1) and quantifies, for each discovered
attribute `g`, the exact amount by which BoN sampling amplifies it. An optional
post-filter further narrows the result set to attributes that humans do *not*
prefer — useful for flagging spurious biases — but the core search target can be
BoN amplification itself, irrespective of human preference.

> Several filesystem locations under `/nfs/data/sohyun/` are hardcoded.

## 1. What this repo does

A reward model assigns higher reward to some visual attributes than others.
Under any reward-maximizing inference procedure (BoN, RLHF), the attributes
correlated with high reward get systematically amplified at the output — even
when the correlation is spurious or unintended. This repo discovers and
quantifies those amplified attributes directly from a fixed image bank, with no
need to re-sample.

This repo answers: **which visual attributes does a given reward model amplify
under BoN?** Optionally, the discovered set can be restricted
to attributes that humans do not actually prefer (the "spurious bias" subset),
but that restriction is optional — the core search target can be purely BoN
amplification.

Concrete artifacts produced by the pipeline:

- A ranked JSON list of binary visual attributes `g` together with their
  measured BoN amplification `Â(g)` at a chosen `N` (default `N=2`, easily
  overridable).
- Prevalence curves for each attribute at `N = 1, 2, 4, 8, 16, 32, 64, 128`,
  showing how the BoN winner's probability of exhibiting the attribute scales
  with the sampling budget.
- Per-prompt diagnostics (R², residual norms) that indicate whether the current
  attribute pool explains the observed reward quantile variation.

The pipeline operates on a fixed image bank generated by **FLUX.1-dev** over
**MJHQ-30K** prompts (clustered into topics under
[clustering/output/mjhq](clustering/output/mjhq)), then scored by a chosen
reward model.

## 2. Algorithmic idea

### 2.1 Setup

- Prompt `x`, image (response) `y`, reward model `r(x, y)`, base policy
  `π_base(·|x)` (here: FLUX.1-dev).
- A binary visual attribute is a function `g(x, y) ∈ {0, 1}` (e.g. "warm color
  cast", "centered subject").
- Best-of-N: draw `y_1, …, y_N ~ π_base(·|x)` i.i.d., then output
  `y* = argmax_i r(x, y_i)`. Call the resulting conditional distribution
  `π^r_N(·|x)`.

The central quantity is the **reward quantile under the base policy**:

```
U_x(y) = P_{y' ~ π_base(·|x)}( r(x, y') ≤ r(x, y) )  ∈ [0, 1].
```

### 2.2 BoN as a reweighted base distribution

By symmetry / independence of the `N` draws, for any test function `g`:

```
E_{y ~ π^r_N(·|x)}[ g(x, y) ]  =  N · E_{y ~ π_base(·|x)}[ g(x, y) · U_x(y)^(N-1) ].
```

Equivalently, BoN reweights `π_base` by the (un-normalized) weight
`N · U_x(y)^(N-1)`.

### 2.3 Theorem 3 in https://arxiv.org/html/2602.01002v1 — the BoN amplification identity

Because `U_x(y)` is uniform on `[0, 1]` when `y ~ π_base`, we have
`E_{π_base}[U_x^(N-1)] = ∫₀¹ u^(N-1) du = 1/N`. Subtracting `E_{π_base}[g]`
from both sides of §2.2 and rearranging:

```
E_{π^r_N}[g] - E_{π_base}[g]  =  N · Cov_{y ~ π_base(·|x)}( g(x, y), U_x(y)^(N-1) )
```

The **search target** for a single attribute is the prompt-averaged amplification:

```
A(g)  :=  N · E_x[ Cov_{y ~ π_base(·|x)}( g(x, y), U_x(y)^(N-1) ) ].
```

Both the sign and the magnitude of BoN's effect on `g` are determined by this
one covariance.

### 2.4 Empirical estimator on a fixed image bank

We never need to *re-sample*. Given:

- prompts `x ∈ X`, each with `M_x` baseline images `y_{x,1}, …, y_{x,M_x}`,
- rewards `r_{x,i} = r(x, y_{x,i})`,
- attribute labels `g_{x,i} ∈ {0,1}` provided by the VLM detector,

define the empirical quantile and its BoN exponent:

```
U_{x,i}      =  #{ j : r_{x,j} ≤ r_{x,i} } / M_x
U_{x,i}^(N-1) ← raise to the (N-1) power
```

The plug-in estimator of `A(g)` is:

```
Â(g)  =  ( N / |X| ) · Σ_{x ∈ X}  Cov_i( g_{x,i}, U_{x,i}^(N-1) )
```

Because `g` is Bernoulli, the within-prompt covariance decomposes cleanly:

```
Cov_x( g, U^(N-1) )  =  p_1(x) · p_0(x) · ( E[U^(N-1) | g=1, x] - E[U^(N-1) | g=0, x] ),
```

where `p_1(x) = (1/M_x) Σ_i g_{x,i}` and `p_0(x) = 1 - p_1(x)`. Prompts where
the attribute is absent or saturated (`p_1 ∈ {0, 1}`) contribute zero.

### 2.5 The discovery loop

The codebase implements three SETUP → EVALUATE → EXPAND cycles. Defaults run
the **`bon_amplified`** mode:

- **SETUP.** Load the baseline image bank + reward scores. The
  `InitialPlanner` shows a VLM (default `openai/gpt-5.2`) seed prompts and
  images, and asks it to propose binary visual attributes. *(Optional)* An
  `AttributeUndesirabilityFilter` further restricts the pool to candidates
  plausibly *undesirable* to humans. Note: currently, it's only implemented to find attributes that humans do not prefer.
- **EVALUATE.** For each candidate `g`, the detector (default
  `Qwen/Qwen3.5-9B` via vLLM) labels all images, then `Â(g)` is computed
  in closed form via §2.4. Candidates below threshold `tau` (default `0`) or outside
  prevalence band `[p1_min, p1_max]` are pruned.
- **EXPAND.** For each prompt, fit `U^(N-1) ≈ Σ_k w_k g_k` (per-prompt
  OLS over the retained pool) and compute residuals `e_{x,i}`. The
  `BonResidualProposer` then mines new candidates by showing the VLM
  **P+ images** (top residual) vs **P- images** (bottom residual) and
  asking what visual feature distinguishes the two sets. New candidates are
  validated by direct `Â`-computation before admission.

The loop terminates when no new candidate clears the admission threshold or
after `evolution.n_steps` iterations.

## 3. Pipeline at a glance

```
        ┌────────────────────────────────────┐
        │  MJHQ-30K prompts                   │
        │  (clustering/output/mjhq)           │
        └────────────────────────────────────┘
                       │
                       ▼
   ┌────────────────────────────────────────────────────────┐
   │ baselines/generate_mjhq.sh                              │
   │   FLUX.1-dev,  M=128 images / prompt                    │
   │   →  topic_<i>/black-forest-labs-FLUX.1-dev/manifest.json
   └────────────────────────────────────────────────────────┘
                       │
                       ▼
   ┌────────────────────────────────────────────────────────┐
   │ baselines/score_baselines.sh                            │
   │   reward_model ∈ { imagereward | pickscore | hpsv3 }    │
   │   →  per-image rewards written back into manifest.json  │
   └────────────────────────────────────────────────────────┘
                       │
                       ▼
   ┌────────────────────────────────────┐    ┌─────────────────────────────────┐
   │ serve_model.sh                      │ ◀──┤ Qwen3.5-9B VLM detector (vLLM)  │
   │   http://localhost:8000/v1          │    │  (one terminal, runs until ^C)  │
   └────────────────────────────────────┘    └─────────────────────────────────┘
                       │
                       ▼
   ┌────────────────────────────────────────────────────────┐
   │ search/run.sh                                           │
   │   SETUP → EVALUATE → EXPAND  (n_steps iterations)       │
   │   → outputs/search/<timestamp>/results.json             │
   └────────────────────────────────────────────────────────┘
                       │
                       ▼
   ┌────────────────────────────────────────────────────────┐
   │ bon/run.sh                                              │
   │   prevalence curves at N = 1, 2, 4, 8, 16, 32, 64, 128  │
   │   → outputs/bon/<timestamp>/                            │
   └────────────────────────────────────────────────────────┘
                       │
                       ▼
              analysis/  (top_attrs, per_prompt_r2, ...)
```

`baselines/` is run **once per (RM, topic)** to populate the image bank. After
that, `search/` and `bon/` are the iterative loop.

## 5. Requirements & setup

### 5.1 Python environments

The project uses **two separate venvs** because FLUX's CUDA/torch pin
conflicts with the looser pin needed for vLLM / OpenAI clients:

1. **Baselines venv** — for FLUX generation only.
   ```bash
   bash baselines/setup_env.sh --venv-dir /nfs/data/sohyun/venvs/baselines
   # Verifies with: from diffusers import FluxPipeline
   ```
   Pinned to `torch==2.5.1+cu124` (see [baselines/requirements.txt](baselines/requirements.txt)).

2. **Main venv** — for `search/`, `bon/`, scoring, and any client code.
   There is no central `requirements.txt`; the main packages (install with
   pip or uv into `.venv`):
   ```bash
   uv venv .venv --python 3.10
   uv pip install --python .venv/bin/python \
       loguru pyyaml rich wandb pillow safetensors huggingface-hub datasets \
       scikit-learn openai anthropic numpy pandas tqdm \
       torch torchvision diffusers transformers accelerate \
       imagereward
   # PickScore/HPSv3 are loaded from HF on first use.
   ```

3. **vLLM venv** *(pre-configured)* — activated automatically by
   `serve_model.sh` from `/nfs/data/sohyun/venvs/vllm-qwen35`. Uses
   `vllm==0.19.1` on `torch==2.10.0+cu128`. You do not need to recreate this
   unless moving to a new host.

### 5.2 API keys (`.env`)

Create `.env` in the repo root (already in `.gitignore`):

```env
OPENAI_API_KEY=sk-...        # planner / filter / proposer / judge default to gpt-5.2 via OpenAI
OPENROUTER_API_KEY=...       # optional, for routing some calls
ANTHROPIC_API_KEY=...        # optional, for Claude-based planners
```

**Important**: if you set `models.detector.model` to a hosted model
(e.g. `openai/gpt-4o-mini`) instead of `Qwen/Qwen3.5-9B`, you do **not** need
to run `serve_model.sh`.

### 5.3 Hardcoded paths (edit for a new host)

These appear across config defaults and shell scripts:

| Path | Used by |
| --- | --- |
| `/nfs/data/sohyun/models` | HF model cache (`HF_HOME`) |
| `/nfs/data/sohyun/tmp` | `TMPDIR` for FLUX + vLLM |

## 6. Running the pipeline

All paths below are relative to the repo root.

### 6.1 Generate baseline images — `baselines/generate_mjhq.sh`

```bash
bash baselines/generate_mjhq.sh \
    --gpus 0,1,2,3 \
    --topics 0 1 2 \
    --python /nfs/data/sohyun/venvs/baselines/bin/python
```

- One Python process per GPU; each process handles **all** requested topics in
  sequence (the FLUX pipe stays resident on its GPU across topics — no
  per-topic reload).
- `--gpus` is required. `--topics` defaults to `0 1 … 9`.
- Other flags: `--model_id` (default `black-forest-labs/FLUX.1-dev`),
  `--output_dir` (default `/nfs/data/sohyun/projects/t2i-rm-bias/data/baselines/mjhq`),
  `--images_per_prompt` (default `128`), `--image_width/--image_height`
  (default `512`).
- Output per topic: `<OUTPUT_DIR>/topic_<i>/black-forest-labs-FLUX.1-dev/manifest.json`
  containing per-prompt lists of `(image_path, seed)`.
- Logs per shard at `<OUTPUT_DIR>/logs/gpu<i>.log`.

### 6.2 Score images — `baselines/score_baselines.sh`

```bash
bash baselines/score_baselines.sh \
    --gpu 0 \
    --topics 0 1 2 \
    --output_dir /nfs/data/sohyun/projects/t2i-rm-bias/data/baselines/mjhq \
    --reward_model imagereward
```

- **Pass `--output_dir` matching the generation output** — the script's
  default differs from `generate_mjhq.sh`'s default (see Troubleshooting).
- `--reward_model ∈ { imagereward, pickscore, hpsv3 }`. Re-run with a
  different `--reward_model` to add another score column; existing scores
  for the same RM are skipped.
- For a single manifest at a non-standard path:
  ```bash
  bash baselines/score_baselines.sh --gpu 0 \
      --manifest /path/to/manifest.json --reward_model pickscore
  ```
- Scores are written back into the same `manifest.json` (in place).

### 6.3 Serve the local Qwen3.5-9B VLM detector — `serve_model.sh`

In a separate terminal:

```bash
bash serve_model.sh 1,2,3,4         # GPUs to use; defaults to "1,2,3,4"
# vLLM serves at http://localhost:8000/v1
```

Verify it is up before running search/BoN:

```bash
curl -s http://localhost:8000/v1/models | head
```

The server stays up until you `^C` it. If you don't want to host a local VLM,
switch the detector model in the config to `openai/gpt-4o-mini` (see §7) and
skip this step.

### 6.4 Run the attribute search — `search/run.sh`

```bash
bash search/run.sh 0 \
    "data.topic_ids=[0]" \
    "data.baseline_manifest=/nfs/data/sohyun/projects/t2i-rm-bias/data/baselines/mjhq/topic_0/black-forest-labs-FLUX.1-dev/manifest.json" \
    "models.reward_model.name=imagereward"
```

- First positional arg = `CUDA_VISIBLE_DEVICES`. Everything after is forwarded
  as YAML dot-path overrides.
- Default config (currently): [search/configs/bon_amplified_partial.yaml](search/configs/bon_amplified_partial.yaml).
  Other configs in [search/configs/](search/configs/):
  - `bon_amplified.yaml` — full BoN-amplified mode (no FLUX).
  - `baseline_pairs.yaml` — regression-over-pairs mode (no FLUX).
  - `default.yaml` — legacy FLUX-edit mode.
  Swap by editing the `python -m search.main --config ...` line in
  [search/run.sh](search/run.sh).
- Output: `outputs/search/<timestamp>/results.json` with the retained pool
  and per-step diagnostics. W&B logging is on by default
  (`logging.wandb.enabled=true`, project `t2i-rm-bias-bon-amplified`).
- Common overrides:
  ```bash
  bash search/run.sh 0 \
      "bon_amplified.N=4" \
      "evolution.n_steps=5" \
      "models.detector.model=openai/gpt-4o-mini" \
      "models.detector.vllm_base_url=null"          # use cloud API instead of local vLLM
  ```

### 6.5 BoN prevalence analysis — `bon/run.sh`

```bash
bash bon/run.sh 1 \
    "attributes.search_results_path=outputs/search/<timestamp>/results.json" \
    "data.baseline_manifest=/nfs/data/sohyun/projects/t2i-rm-bias/data/baselines/mjhq/topic_0/black-forest-labs-FLUX.1-dev/manifest.json" \
    "data.topic_ids=[0]" \
    "models.reward_model.name=pickscore"
```

- First positional arg = `CUDA_VISIBLE_DEVICES`. Rest are dot-path overrides.
- Config: [bon/configs/default_mjhq.yaml](bon/configs/default_mjhq.yaml).
  Alternative: [bon/configs/random_baseline.yaml](bon/configs/random_baseline.yaml) for sanity-check
  curves on random attributes.
- Set `data.val_split_size` to **match** the value used by the search run
  that produced `search_results_path` — otherwise the BoN analysis sees a
  different val/train partition.
- Attributes are loaded from `attributes.search_results_path` by default; an
  explicit list `attributes: ["high saturation", "warm tone", ...]` in the
  YAML or CLI overrides that.
- Output: `outputs/bon/<timestamp>/` with prevalence curves per attribute
  across `sampling.n_values = [1, 2, 4, 8, 16, 32, 64, 128]` (configurable),
  averaged over `sampling.n_trials = 100` Monte Carlo trials.

## 7. Configuration

All configs are YAML. Defaults live next to the entry points
([search/configs/](search/configs/), [bon/configs/](bon/configs/)). The CLI accepts
**dot-path overrides** that are merged on top of the YAML:

```bash
bash search/run.sh 0 "evolution.n_steps=5" "data.topic_ids=[0,1]"
```

Most-edited keys per config:

**Search ([search/configs/bon_amplified_partial.yaml](search/configs/bon_amplified_partial.yaml))**

- `bon_amplified.N` — the `N` in `U^(N-1)`. Default `2`.
- `bon_amplified.tau` / `bon_amplified.tau_partial` — `Â(g)` pruning thresholds.
- `bon_amplified.use_p1_filter`, `p1_min`, `p1_max` — drop attributes too rare or too ubiquitous.
- `evolution.n_steps`, `evolution.target_pop_sizes` — EXPAND iterations and top-K admission per step.
- `evaluation.amp_n_prompts`, `amp_n_images_per_prompt` — size of the fixed baseline set used for `Â`.
- `data.topic_ids`, `data.baseline_manifest` — which manifest(s) to use.
- `models.reward_model.name` — which RM to attack.
- `models.detector.model`, `models.detector.vllm_base_url` — local vs hosted detector.

**BoN ([bon/configs/default_mjhq.yaml](bon/configs/default_mjhq.yaml))**

- `sampling.n_values`, `sampling.n_trials` — BoN budgets and MC trials.
- `attributes.search_results_path` — which search run to consume; or set `attributes.attributes: [...]` to pin an explicit list.
- `data.val_split_size` — must match the producing search run.
- `models.reward_model.name` — RM for BoN winner selection.

Both configs validate at startup via the dataclasses in
[search/config.py](search/config.py) and [bon/config.py](bon/config.py) — typos in
override paths cause an immediate, loud failure.

## 8. Outputs & analysis

| Location | Contents |
| --- | --- |
| `outputs/search/<timestamp>/` | `results.json` (retained pool + `Â(g)` per step), per-step metric dumps, plots if any. |
| `outputs/bon/<timestamp>/` | Prevalence curves per attribute, summary JSON, per-prompt diagnostics. |
| `outputs/detection_cache/mjhq/black-forest-labs-FLUX.1-dev.json` | Cached VLM detection labels keyed by `(attribute, image)`. Reused across search and BoN runs. |
| `wandb/` | W&B local run state. Project names: `t2i-rm-bias-bon-amplified`, `t2i-rm-bias-bon`. |

Post-hoc tooling in [analysis/](analysis/) (each is a thin CLI over the run outputs):

- `top_attrs.py` — markdown table of top-`Â(g)` attributes per run; can cross-reference BoN prevalence deltas if a BoN run path is provided.
- `per_prompt_r2.py` — per-prompt R² of the OLS fit across EXPAND iterations; diagnostic for whether the pool is converging.
- `detect_extra_attrs.py` — run the detector on additional attributes against an existing image bank (without re-running search).
- `generate_random_attrs.py` — sample random plausible attributes as a control baseline.
